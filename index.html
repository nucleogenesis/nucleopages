• Snackbars

    Accessiblity Review

  • What is a Snackbar?

    • In Kolibri / KDS

      • Let the user know that something happened following an action

      • Use for non-critical information (prefer modal / notification for critical)

      • Allow a user to take an action in response to the result of an action (ie, undo)

    • Toasts == Banners == Notifications == ...

      • None of those that I researched have "Snackbars"

      • There were suggestions that were similar in nature, but they were always referred to as Banners, Toasts or Notifications

    • Comparing to KDS

      • UX Commonalities

        Generally, our UX maps to most of the suggestions, examples:

        • Actionable messages persist until interacted with

        • Transient "Toast" messages should disappear automatically after a short time

        • Placement is out of the way, but in a noticeable position

      • UX Differences

        • Some suggest allowing at once

        • Iconographic & color-based states to differentiate between a success, warning, failure, etc.

      • Banners

        • I just wanted to note that this was a common alternative in the systems which did not have a similar component.

        • The pattern here is currently being discussed in an open GH Issue on Kolibri

      • a11y Highlights

        • IBM Carbon

          They had two references to WCAG 2.1 standards:

          • We do this

            Don’t use notifications that dismiss on a timer for critical or emergency messages. Some users with disabilities need more time to read or interact with messages and timed actionable toasts may not provide sufficient time. WCAG 2.1 success criterion 2.2.4 (AAA)

          • We don't do this

            Users should be able to manage or limit noncritical notifications. This gives users the control to reduce the number of distractions or disruptions they receive, which is particularly helpful for users with cognitive limitations. WCAG 2.1 success criterion 2.2.3 (AAA)

        • Access Use

          We don't have any aria business in the snackbar

          • Name non-modal dialogs meaningfully. Refer to their visible heading via aria-labelledby, or provide an invisible name via aria-label.

          • Label all dialog controls, including the close button. Each dialog should have a clearly labelled control for closing it. Icon-based buttons for closing [x] must have an accessible name that is read out to screen reader users.

          • Use the correct ARIA roles. Use role="dialog" and aria-modal="false" on the dialog.

        • General Thoughts

          • Focus handling is also important for keyboard users - when a Snackbar steals focus, it needs to return the user to where they were

          • Multiple Snackbars are generally said to be okay, stacking them vertically (but don't overdo it)

          • Use polite to make screen-readers aware without stealing focus (aria-live => polite vs assertive) - polite lets the user know there is a notification, but doesn't steal focus, assertive will force the user to focus the notification

      • Final Thoughts

        • Snackbar lives in Kolibri, not KDS - should this change?

        • We are aiming to implement a Banner similar to those noted in the design systems above, should bear these resources in mind

        • Value of adding stacked Snackbars vs queuing them (which, is what KDS suggests but is not imlemented IIRC)
